# 14. RAG‑Style Reasoning and Contextual Answer Generation (rag_query.py + purge_rag_query.py)

This stage brings the entire retrieval pipeline to life. After the
system can search semantically, this script adds the second half of
Retrieval‑Augmented Generation: **combining retrieved context with a
question and generating an answer grounded strictly in the
documentation**. It also introduces multiple response modes—default,
strict, developer, bullets, and an automatic mode selector—so the system
can adapt the style of the answer to the user’s intent.

## 14.1 Purpose of This Stage

The goal of this stage is to take a user question, retrieve the most
relevant documentation chunks, build a structured context, and generate
an answer using a local LLM (Qwen 2.5 running on Ollama). This ensures
that every answer is grounded in the Azure Event Grid documentation
rather than relying on the model’s general knowledge.

This stage is where the pipeline starts behaving like a real RAG system.

## 14.2 How the RAG Script Works (rag_query.py)

The script orchestrates four major steps: retrieval, context building,
prompt construction, and LLM generation.

**1. Load the retriever dynamically**

Instead of importing the retriever normally, the script loads
6.retriever.py dynamically. This ensures that the RAG script always uses
the correct version of the retriever, even if the file is renamed or
reorganized.

It exposes the search() function, which retrieves the top‑k relevant
chunks.

**2. Build the context**

Once the retriever returns the relevant chunks, the script formats them
into a readable context block. Each block includes:

- the source markdown file

- the documentation path (TOC hierarchy)

- the cleaned text

Blocks are separated with clear dividers so the LLM can distinguish
them.

**3. Choose a prompt mode**

The script supports five modes:

- **default** — a normal grounded answer

- **strict** — refuses to answer if the context does not explicitly
  contain the information

- **developer** — produces a technical explanation with structure and
  references

- **bullets** — produces a concise, structured bullet‑point answer

- **auto** — detects the best mode based on the question

The auto‑mode detector looks for keywords that suggest the user wants:

- a technical explanation

- a list or summary

- a strict context‑only answer

This makes the system more natural and adaptive.

**4. Build the final prompt**

The script inserts:

- the cleaned context

- the user question

- the selected prompt template

Each template instructs the LLM to use **only** the provided context.

**5. Call the local LLM**

The script sends the prompt to a local Ollama server running the Qwen
2.5 model. This keeps the entire RAG process private, offline, and
cost‑free.

The LLM returns a grounded answer based solely on the context.

**6. Return the answer and the retrieved sources**

The function returns:

- the generated answer

- the list of retrieved chunks

This makes it easy to inspect the sources behind every answer.

## 14.3 Why This Stage Matters

This stage is the first time the pipeline produces **human‑readable
answers** grounded in the documentation. It is essential for:

- validating the quality of retrieval

- generating training data

- simulating how the final LoRA model will behave

- ensuring that answers remain faithful to the documentation

The multiple response modes also allow the system to adapt to different
use cases, from technical explanations to concise summaries.

## 14.4 Cleaning Previous Outputs (purge_rag_query.py)

This companion script resets the RAG stage by deleting only the log
file. It does not delete any data or model outputs—just the logs—so you
can run clean tests without leftover traces.

## 14.5 Inputs and Outputs of This Stage

**Inputs**

- the retriever (search() function)

- FAISS index and metadata (indirectly)

- cleaned corpus chunks

- user question

- local LLM (Qwen 2.5 via Ollama)

**Outputs**

- grounded answer generated by the LLM

- list of retrieved chunks

- rag_query.log — detailed log of the process

## 14.6 Role of This Stage in the Pipeline

This stage is the bridge between retrieval and dataset creation. It
demonstrates how the system uses the documentation to answer questions,
and it sets the foundation for generating high‑quality QA pairs in the
next stages. It also provides a preview of how the final LoRA‑trained
model will behave when deployed.
